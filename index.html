Built for the people who fix the equipment

Maintenance and reliability teams don't have time to search a manual page by page while an asset is down. This project turns that search into a single question — and an answer you can verify.

💬

Describe the fault, not the part number

"The crusher's eccentric shaft is vibrating" or "conveyor belt slipping on the drive pulley" — plain language, no need to know internal component codes.

🧭

Get the probable cause and the fix

Every answer names the likely cause and the corrective action, with the exact chain of relationships and the source manual it came from.

❓

Asked for detail, not given a guess

If the question doesn't specify enough (e.g. a component with several possible symptoms), the agent asks which one — instead of picking one at random.

What is a KAG?

KAG stands for Knowledge Augmented Generation — an approach that grounds an LLM's answer in a structured knowledge graph of real entities and relationships, instead of loose retrieved text. Letter by letter:

K

Knowledge

A structured graph of real entities and relationships — equipment, components, symptoms, causes and actions — extracted from the manuals, instead of an unstructured pile of text passages.

A

Augmented

The model's prompt is enriched with the exact path found in that graph — the specific chain of edges connecting the reported symptom to a cause and a fix — not a loose paragraph that merely sounds related.

G

Generation

The model writes the final answer using only that path, under strict rules: cite the exact chain of relationships followed and the manual it came from — never add a relationship that wasn't found in the graph.

The wiring-diagram analogy: a plain AI chat is like a technician working from memory — confident, but occasionally wrong. This system works like someone following the plant's actual wiring diagram: before answering, it traces the real path from symptom to cause to corrective action, and only writes the answer using a path that truly exists in the diagram.

🤖

1 · Extraction

An LLM reads each manual and extracts structured relationships — origin, relation, destination — using only four allowed relation types. Anything that doesn't fit the schema is discarded, not forced in.

🕸️

2 · Graph construction

The relationships become one directed graph. The same entity mentioned in different manuals becomes a single shared node — this is how knowledge from separate documents connects.

🧮

3 · Traversal & generation

A plain algorithm — not the LLM — walks the graph from the identified entity, up to three hops. The LLM only writes the final sentence from the path it's handed.

Equipment → Component → Symptom → Cause → Corrective Action

Every answer is a walk along this chain — never a shortcut across it.

💡 If the graph has no path for a question — or the question is too vague to choose one — the agent says so, or asks for the missing detail, instead of inventing a relationship.

KAG vs. RAG — what each one is for

Both are ways to ground an LLM in your own data instead of its training memory. They differ in what counts as "retrieved evidence" — and that changes what each is good at.

Best for: open-ended document Q&A

Traditional RAG

Splits documents into chunks and retrieves the ones that sound closest to the question. Works great when the answer lives inside a single passage — a contract clause, a paragraph in a report, an FAQ entry.

Struggles when the answer requires connecting facts scattered across different passages — the model has to guess how they relate, which is exactly where it can hallucinate.

Best for: multi-hop, structured reasoning

Knowledge Augmented Generation

Matches the question to a node in a real graph and walks the actual edges from there. Built for domains with explicit cause-and-effect chains — like equipment diagnostics — where the answer is a path, not a passage, and every step needs to be auditable.

Costs more to set up (extraction + graph modeling) and fits a narrower, more structured domain than open-ended RAG — the trade-off for a reasoning chain you can fully trace.

Architecture: from maintenance manual to grounded answer

Every step of the pipeline is a small, readable Python module — built by hand, with no graph-RAG framework in between.

1

Source manuals — data/raw/

Four synthetic maintenance manuals, one per mining asset type, each describing the equipment's components and their known failure modes.

▼
2

Extraction — src/extraction.py

An LLM reads each manual and extracts structured triples — origin, relation, destination — using only four allowed relations. Anything that doesn't fit the schema, or mixes up entity types, is discarded rather than forced in.

▼
3

Graph construction — src/graph_builder.py

The triples become one directed graph (NetworkX) — no external graph database. The same entity mentioned in different manuals becomes a single shared node.

▼
4

Entity linking & specificity check — src/graph_query.py

An LLM maps the question to one real node, scoped strictly to the equipment mentioned. A deterministic check then verifies the question was specific enough — if not, the agent asks for the missing detail instead of picking an option at random.

▼
5

Deterministic traversal — src/graph_query.py

Plain graph traversal — not the LLM — walks up to three hops from the identified entity, following the real causal chain: symptom → cause → corrective action.

▼
6

Grounded answer — src/answer_builder.py

The LLM writes the final answer using only the traced path, citing the exact chain of relationships and the source manual — never adding a relationship that wasn't found.

▼
7

Interface — app.py (Streamlit)

A chat interface with separate conversations (create, rename, delete), a live animation tracing the graph nodes visited while the answer is prepared, and the full reasoning path shown below each response.

Four synthetic manuals, one mining domain

Each manual is an independent source document — its own components, symptoms, causes and corrective actions — before extraction unifies all four into a single graph.

⚠️ All data below is synthetic. These four manuals, and every component, symptom, cause and corrective action inside them, were written to simulate realistic maintenance documentation — none of it was extracted from a real manufacturer's manual or a real mining operation.
🔧

Vibrating Screen — PVI-1000

7 components covered

Classifies crushed ore by particle size before secondary crushing or grinding. Covers the drive motor, eccentric, eccentric bearings, support springs, screen mesh, vibrating structure and rubber dampers.

Example: drive motor vibration traced to motor–eccentric misalignment; torn screen mesh traced to oversized rock impact.

📦

Belt Conveyor — TC-2000

9 components covered

Long-distance continuous transport of crushed ore between processing stages. Covers the drive motor, gearbox, drive and return pulleys, carrying and return idlers, belt, take-up and cleaning scrapers.

Example: a torn belt traced to a metal contaminant puncturing it; belt slippage on the drive pulley traced to low belt tension.

🚛

Haul Truck — CFE-400T

7 components covered

Hauls ore and waste rock between the pit and the crushing plant or dump sites. Covers the diesel engine, transmission, brakes, tires, dump body, hoist hydraulics and hydro-pneumatic suspension.

Example: engine overheating traced to low coolant or a blocked radiator; irregular tire wear traced to incorrect tire pressure.

⚙️

Jaw Crusher — BM-1200

12 components covered

Primary crushing stage — the first size reduction of run-of-mine ore. Covers the fixed and moving jaw plates, eccentric shaft, flywheel, pitman, main bearings, tension springs, toggle plate, drive belts and lubrication system.

Example: a broken tension spring traced to overload from an unbreakable object in the crushing chamber; toggle plate wear traced to insufficient lubrication.
